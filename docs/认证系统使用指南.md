# 认证系统使用指南

本文说明 Phase 11 认证系统的使用方式。当前系统支持两种模式：

- **本地模式**：关闭认证，继续使用默认用户 `default`。
- **部署模式**：启用 GitHub OAuth，只有白名单中的 GitHub 用户可以访问应用。

认证开关在后端配置中控制。前端会自动读取后端认证状态，并决定是否要求登录。

## 快速结论

本地开发时保持默认配置即可：

```yaml
# atri/config/auth.yaml
enabled: false
```

公开部署前启用认证：

```yaml
# atri/config/auth.yaml
enabled: true
```

启用认证后，访问前端 `/` 会自动跳转到 `/login`。用户必须完成 GitHub 登录后才能使用主页、设置页、聊天、WebSocket 等功能。

## 本地模式

本地模式适合个人开发和单机部署。

1. 确认后端认证关闭：

```yaml
# atri/config/auth.yaml
enabled: false
```

2. 启动后端和前端。

3. 打开前端页面。

此时不会跳转登录页。后端会使用 `default` 作为用户 ID，并保持原来的单用户聊天数据路径。

## 部署模式

部署模式适合公网服务器或多人可访问环境。

### 1. 创建 GitHub OAuth App

在 GitHub 创建 OAuth App：

1. 打开 GitHub **Settings**。
2. 进入 **Developer settings**。
3. 进入 **OAuth Apps**。
4. 创建新的 OAuth App。

本地开发示例：

| 字段 | 值 |
| --- | --- |
| Homepage URL | `http://localhost:5173` |
| Authorization callback URL | `http://localhost:8430/api/auth/callback` |

生产环境示例：

| 字段 | 值 |
| --- | --- |
| Homepage URL | `https://YOUR_FRONTEND_DOMAIN` |
| Authorization callback URL | `https://YOUR_BACKEND_DOMAIN/api/auth/callback` |

注意：GitHub OAuth App 的 callback URL 是**后端**地址。后端拿到 GitHub 授权结果后，再跳转到前端 `/auth/callback`。

### 2. 配置后端环境变量

在 `atri/.env` 中配置：

```env
JWT_SECRET_KEY=REPLACE_WITH_A_LONG_RANDOM_SECRET
GITHUB_CLIENT_ID=YOUR_GITHUB_CLIENT_ID
GITHUB_CLIENT_SECRET=YOUR_GITHUB_CLIENT_SECRET
```

要求：

- `JWT_SECRET_KEY` 必须是长随机字符串。
- 不要把真实 `.env` 提交到 Git。
- `GITHUB_CLIENT_ID` 和 `GITHUB_CLIENT_SECRET` 来自 GitHub OAuth App。

### 3. 配置后端认证开关

编辑 `atri/config/auth.yaml`：

```yaml
enabled: true

jwt:
  secret_key: ${JWT_SECRET_KEY}
  algorithm: HS256
  expire_days: 7

github:
  client_id: ${GITHUB_CLIENT_ID}
  client_secret: ${GITHUB_CLIENT_SECRET}
  callback_url: http://localhost:8430/api/auth/callback
  scope: read:user

frontend:
  callback_url: http://localhost:5173/auth/callback
  login_url: http://localhost:5173/login

whitelist:
  users:
    - YOUR_GITHUB_USERNAME
```

生产环境需要把 URL 改成实际域名：

```yaml
github:
  callback_url: https://YOUR_BACKEND_DOMAIN/api/auth/callback

frontend:
  callback_url: https://YOUR_FRONTEND_DOMAIN/auth/callback
  login_url: https://YOUR_FRONTEND_DOMAIN/login
```

JWT 生命周期固定使用 `expire_days: 7`，也就是 7 天。

### 4. 配置白名单

只有 `whitelist.users` 中的 GitHub 用户名可以登录。

```yaml
whitelist:
  users:
    - JuyaoHuang
    - another-user
```

白名单匹配不区分大小写。建议填写 GitHub profile 上显示的 `login` 用户名。

### 5. 配置前端地址

本地开发使用 `atri-webui/.env.development`：

```env
VITE_API_BASE_URL=http://localhost:8430
VITE_WS_URL=ws://localhost:8430/ws
```

生产环境按部署域名配置：

```env
VITE_API_BASE_URL=https://YOUR_BACKEND_DOMAIN
VITE_WS_URL=wss://YOUR_BACKEND_DOMAIN/ws
```

如果前端和后端同域反向代理，也可以使用相对 API 代理方案，但要确保 `/api` 和 `/ws` 都能转发到后端。

## 登录流程

启用认证后，流程如下：

1. 用户访问 `/`。
2. 前端路由守卫调用 `/api/auth/status`。
3. 如果认证开启且没有有效 token，前端跳转 `/login?redirect=/`。
4. 用户点击 **Continue with GitHub**。
5. 前端调用 `/api/auth/login` 获取 GitHub 授权地址。
6. 用户在 GitHub 授权。
7. GitHub 回调后端 `/api/auth/callback`。
8. 后端读取 GitHub 用户信息并检查白名单。
9. 白名单通过后，后端生成 JWT。
10. 后端跳转前端 `/auth/callback?token=...`。
11. 前端保存 token 到 LocalStorage，并回到原目标页面。

## 前端行为

启用认证后：

- `/login` 和 `/auth/callback` 是公开页面。
- `/`、`/settings` 和其他业务页面都需要登录。
- 已登录用户访问 `/login` 会回到目标页面。
- API 返回 `401` 时，前端清除 token 并跳转登录页。
- 登录页会显示 “Login has expired. Please sign in again.”。
- WebSocket 连接会自动追加 `?token=JWT_TOKEN`。

关闭认证时：

- 不会强制跳转登录页。
- 前端继续使用本地用户资料。
- 后端使用 `default` 用户。

## 账号设置页

账号设置页路径：

```text
/settings/account
```

该页面显示：

- 当前认证模式。
- GitHub 用户名和头像。
- 登录时间。
- 登出按钮。
- 本地显示名和本地头像文件名。

本地显示名和头像仍用于本地模式，以及聊天消息中的用户展示。

## 后端认证接口

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| `GET` | `/api/auth/status` | 返回认证是否启用 |
| `GET` | `/api/auth/login` | 返回 GitHub OAuth 授权地址 |
| `GET` | `/api/auth/callback` | GitHub OAuth 回调 |
| `GET` | `/api/auth/me` | 返回当前用户 |
| `POST` | `/api/auth/logout` | 前端登出用，服务端返回成功 |

启用认证后，除认证接口、静态资源、健康检查和文档接口外，后端 HTTP 路由都需要 `Authorization` header：

```http
Authorization: Bearer YOUR_JWT_TOKEN
```

WebSocket 使用查询参数：

```text
wss://YOUR_BACKEND_DOMAIN/ws?token=YOUR_JWT_TOKEN
```


手动验收：

1. 设置 `enabled: true`。
2. 填写 `.env` 中的 OAuth 和 JWT 配置。
3. 把你的 GitHub 用户名加入白名单。
4. 启动后端和前端。
5. 打开前端 `/`。
6. 确认自动跳转 `/login`。
7. 点击 GitHub 登录。

## 常见问题

### 启用认证后后端启动失败

可能原因：

- `JWT_SECRET_KEY` 未设置。
- `GITHUB_CLIENT_ID` 未设置。
- `GITHUB_CLIENT_SECRET` 未设置。
- `auth.yaml` 中仍是 `${...}` 占位符。

解决方式：

1. 检查 `atri/.env`。
2. 确认后端启动时加载了 `.env`。
3. 确认 `config/auth.yaml` 中引用了正确变量。

### GitHub 回调后登录失败

可能原因：

- GitHub OAuth App 的 callback URL 配错。
- `github.callback_url` 与 GitHub OAuth App 不一致。
- `frontend.callback_url` 不是前端实际域名。
- GitHub 用户名不在白名单。

解决方式：

1. GitHub OAuth App callback 使用后端 `/api/auth/callback`。
2. `frontend.callback_url` 使用前端 `/auth/callback`。
3. 检查 `whitelist.users`。

### 访问主页一直跳登录页

可能原因：

- Token 不存在。
- Token 已过期。
- Token 对应用户不在白名单。
- 浏览器 LocalStorage 中保存了旧 token。

解决方式：

1. 重新登录。
2. 清理浏览器 LocalStorage 中的 `atri_auth_token`。
3. 确认白名单配置。

### WebSocket 连接被关闭

启用认证后，WebSocket 缺少 token 或 token 无效时，后端会拒绝连接。

检查：

- 前端是否已登录。
- `VITE_WS_URL` 是否指向后端 `/ws`。
- 浏览器开发者工具中 WebSocket URL 是否带有 `token` 查询参数。

### 本地不想登录

把后端配置改回：

```yaml
enabled: false
```

重启后端后，前端会恢复本地免登录行为。

## 安全注意事项

- 生产环境必须使用 HTTPS 和 WSS。
- 不要提交真实 `.env`。
- `JWT_SECRET_KEY` 泄露后，需要更换密钥并让所有用户重新登录。
- 白名单越短越安全。
- 公网部署时必须启用认证。
